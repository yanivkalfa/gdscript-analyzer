# Ecosystem & Release Tooling for `gdscript-analyzer`

**A "Roslyn for Godot": building the OSS ecosystem/tooling FIRST — repo structure, CI, git flow, versioning, changelogs, docs, governance, licensing, and an automated "stay synced with Godot" process.**

Research note — compiled 2026-06-22. Every load-bearing claim is cited inline with a URL. Versions are snapshots as of June 2026; the *structural* patterns (workspace shapes, CI matrices, release flows, governance files) are stable across releases unless noted.

Cross-references the sibling notes in this series: distribution/packaging mechanics are in `01-rust-distribution-tooling.md`; the API-sync *content* (what the diff means, codegen) is in `03-godot-api-sync.md`; analyzer crate layering is in `06-analyzer-architecture.md`; the wasm/napi binding details are in `08-wasm-web-and-bindings.md`. This note focuses on the **ecosystem and release-engineering wrapper** around all of that.

## 0. TL;DR recommendation

| Concern | Choice | Why |
|---|---|---|
| **Repo layout** | Flat `crates/*` (folder == crate name) under a **virtual workspace root**; `bindings/{node,wasm}` for the napi/wasm surfaces; `xtask/` for build automation; `fixtures/`, `docs/`, `.github/` | matklad's "Large Rust Workspaces" + rust-analyzer/swc split, with napi-rs per-platform `npm/` packaging |
| **Build automation** | **cargo-xtask** (`cargo xtask <cmd>` via `.cargo/config.toml` alias) | Pure-Rust, cross-platform, no `make`/`bash`/`python` dependency; used by Cargo, rust-analyzer, helix |
| **CI** | GitHub Actions: `fmt` → (`clippy`, `test` 3-OS matrix, `msrv`, `coverage`) gated behind it; separate napi cross-compile matrix + wasm job; `Swatinem/rust-cache` everywhere | napi-rs package-template + oxc reusable workflows |
| **Cross-compile** | napi-rs CLI `--use-napi-cross` (zig, gnu/arm) + `-x`/cargo-zigbuild (musl); native macOS/Windows runners | The napi-rs v3 default; lighter than swc's Docker-per-target |
| **Crate release** | **release-plz** (Release-PR: version bump + git-cliff changelog + `cargo-semver-checks` + crates.io publish via OIDC) | Automated, Conventional-Commits-driven, the modern cargo-workspace standard |
| **npm release** | **Changesets** (`.changeset/*.md` intent files → "Version Packages" PR → `npm publish --provenance`) | What Biome uses; the JS-ecosystem standard |
| **Version sync** | **One shared version** across crates.io + npm; release-plz is the source of truth, a one-line sync step copies it into each `package.json` before npm publish | Cargo & npm interpret `0.x` carets identically, so one number behaves the same on both registries |
| **CLI binaries** | **cargo-dist / `dist`** keyed off the release tag (only if/when a standalone CLI ships) | Complementary to release-plz; builds artifacts + installers on the GitHub Release |
| **Changelog format** | **Keep a Changelog 1.1.0** headings, generated by git-cliff (inside release-plz) from **Conventional Commits 1.0.0** | Human-readable + machine-derivable |
| **Git flow** | **Trunk-based**: protected `main`, short-lived branches, **squash-merge with PR-title-as-conventional-commit**, GitHub **merge queue** | rust-analyzer's bors/"not rocket science" rule, expressed in GitHub-native form |
| **Versioning** | **SemVer 2.0.0** with the **Cargo 0.x reading** (`0.Y.0` = breaking, `0.y.Z` = compatible); start at `0.1.0`, lockstep across all crates+packages | Safe, predictable, identical semantics on both registries |
| **Docs** | **mdBook** guide (GitHub Pages starter workflow) + **docs.rs** API docs (`--cfg docsrs`); README with crates.io/docs.rs/CI/npm badges | rust-analyzer manual + docs.rs, the Rust-first split |
| **Governance** | Standard GitHub community-health files + **YAML issue forms**; **lightweight in-repo ADRs now**, defer a separate RFC repo; enable Private Vulnerability Reporting + `cargo-audit` CI | rust-analyzer triage + Bevy's "RFCs are a tool, not a burden" philosophy |
| **License** | **`MIT OR Apache-2.0`** dual license + both LICENSE files; `THIRD-PARTY-NOTICES.md` for Godot + tree-sitter-gdscript (both MIT) | The Rust ecosystem norm; all ingested inputs are MIT → clean, attribution-only obligation |
| **Godot sync** | Scheduled Action (`schedule` + `workflow_dispatch`): resolve latest Godot tag via `gh api`, fetch `extension_api.json`, diff against the committed copy, open/update a PR via `peter-evans/create-pull-request` | GitHub-native, idempotent, surfaces the API delta in the PR body |

The closest single template is **oxc** (Rust core + napi + wasm + npm), with **rust-analyzer**'s library-layering and `xtask` discipline and **Biome**'s Changesets-for-npm split.

## 1. Cargo workspace layout & repo structure

### 1.1 What the exemplars actually do

| Repo | `crates/` dir | Native (napi) | wasm | Automation | npm packages |
|---|---|---|---|---|---|
| **rust-analyzer** | flat `crates/*` (~30) | n/a | n/a | **`xtask/`** | n/a |
| **Biome** | `crates/biome_*` | n/a | `crates/biome_wasm` → 3 wasm pkgs | **`xtask/`** + `justfile` | `packages/@biomejs/` (`biome` + 8 `cli-*` + 3 `wasm-*`) |
| **oxc** | `crates/oxc_*` | top-level **`napi/`** | from napi crate (`wasm32-wasi`) | custom **`tasks/`** | top-level **`npm/`** |
| **swc** | `crates/swc_*` (~150+) | **`bindings/binding_core_node`** | **`bindings/binding_core_wasm`** | **`xtask/`** + `tools/` | `packages/` |
| **Ruff** | `crates/ruff_*` + `ty_*` | maturin `bin` | `ruff_wasm`, `ty_wasm` | `ruff_dev` crate | `python/ruff` wheel |
| **ripgrep** | `crates/*` | none | none | `ci/` + `scripts/` shell | none |
| **tokio** | members at root | none | none | none | none |

Sources: rust-analyzer architecture <https://github.com/rust-lang/rust-analyzer/blob/master/docs/book/src/contributing/architecture.md>; Biome packages <https://github.com/biomejs/biome/tree/main/packages/%40biomejs>; oxc tree <https://github.com/oxc-project/oxc> + napi <https://github.com/oxc-project/oxc/tree/main/napi> + tasks <https://github.com/oxc-project/oxc/tree/main/tasks>; swc bindings <https://github.com/swc-project/swc/tree/main/bindings>; Ruff crates <https://github.com/astral-sh/ruff/tree/main/crates>; ripgrep <https://github.com/BurntSushi/ripgrep/tree/master/crates>; tokio Cargo.toml <https://github.com/tokio-rs/tokio/blob/master/Cargo.toml>.

### 1.2 Layout philosophy (matklad, "Large Rust Workspaces")

The canonical rationale — <https://matklad.github.io/2021/08/22/large-rust-workspaces.html>:

- **Flat `crates/` beats nested hierarchies** for 10k–1M LOC. Cargo's crate namespace is already flat (a folder tree just adds a second, divergent system); `ls ./crates` gives an instant overview ("even comparatively large lists are easier to understand at a glance than even small trees"); flat layouts don't rot into mostly-empty/catch-all dirs.
- **Name each crate exactly as its folder** (`hir_def`, `ide_completion`) — trivial navigation and renames.
- **Make the workspace root a virtual manifest** (only `[workspace]`, no `[package]`), so the root holds no source.
- **Use `cargo xtask` for automation** instead of ad-hoc scripts.

### 1.3 The cargo-xtask pattern

**What it is:** xtask is "a way to add free-form automation to a Rust project, a-la `make`, `npm run` or bespoke bash scripts" — <https://github.com/matklad/cargo-xtask/blob/master/README.md>. You add a regular binary crate named `xtask` to the workspace; it dispatches on its first positional arg and runs arbitrary Rust (codegen, dist, release, install).

**Why over Make/shell** (both from the README): (1) "it doesn't require any other binaries besides `cargo` and `rustc`, it fully bootstraps from them" — no `make`/`bash`/`python` dependency; (2) "Unlike bash, it can more easily be cross platform, as it doesn't use the shell" — which matters a lot for a Windows + macOS + Linux library.

**Wiring** — a single cargo alias:

```toml
# .cargo/config.toml
[alias]
xtask = "run --package xtask --"
```

so `cargo xtask <cmd>` works from any subdirectory. Adopters named in the README: **Cargo itself, rust-analyzer, helix, bootc**. rust-analyzer confirms the same role: "We use cargo to compile rust code, but there are also various other tasks, like release management or local installation. They are handled by Rust code in the xtask directory" (architecture.md, above).

**For `gdscript-analyzer`, xtask owns:** codegen from `extension_api.json` (see `03-godot-api-sync.md`), parser-test golden-tree regeneration, the napi/wasm `dist` orchestration, and local-install helpers.

### 1.4 napi-rs per-platform npm packaging (recap; full detail in `01` and `08`)

A multi-platform napi-rs package emits a **main JS package + one binary package per platform/arch**, wired via `optionalDependencies` + each platform package's own `os`/`cpu` so the package manager installs only the matching one — <https://napi.rs/docs/deep-dive/release>. The generated `index.js` switches on `process.platform`/`process.arch`; on Linux x64 it tries both `-gnu` and `-musl`. Reference repo shape: <https://github.com/napi-rs/package-template/tree/main>. This is identical to the swc/Biome loader idiom documented in `01-rust-distribution-tooling.md`.

### 1.5 Recommended directory tree

Blends rust-analyzer/matklad's **flat `crates/` + virtual root + `xtask/`** with swc's clean **`bindings/` split** and napi-rs's **per-platform `npm/`** packaging.

```
gdscript-analyzer/
├── Cargo.toml                  # virtual workspace manifest (no [package]) — matklad
├── Cargo.lock                  # committed (workspace has binaries)
├── rust-toolchain.toml         # pin toolchain; feeds rust-cache key
├── rustfmt.toml  clippy.toml
├── deny.toml                   # cargo-deny: license allow-list + advisory check
├── cliff.toml / release-plz.toml
├── .changeset/                 # Changesets intent files (npm side)
│   └── config.json
├── .cargo/
│   └── config.toml             # [alias] xtask = "run --package xtask --"
├── crates/                     # FLAT layout, crate name == folder name
│   ├── gdscript-syntax/        # lossless syntax tree (rowan-style) + parser
│   ├── gdscript-core/          # salsa/base_db: inputs, interning, query infra
│   ├── gdscript-hir/           # name resolution, types, semantic model
│   ├── gdscript-ide/           # IDE features (completion/hover/diagnostics), LSP-agnostic

│   ├── gdscript-lsp/           # LSP server binary wrapping `ide`
│   └── gdscript-cli/           # `gdscript-analyzer` CLI (check/format/lint)
├── bindings/                   # swc-style split of the two distribution surfaces
│   ├── node/                   # napi-rs crate (#[napi]) → native Node addon
│   │   ├── Cargo.toml  build.rs  src/lib.rs
│   │   ├── index.js  index.d.ts   # generated by @napi-rs/cli
│   │   └── npm/                    # per-platform packages (optionalDependencies)
│   │       ├── darwin-x64/  darwin-arm64/
│   │       ├── linux-x64-gnu/  linux-x64-musl/  linux-arm64-gnu/  linux-arm64-musl/
│   │       └── win32-x64-msvc/  win32-arm64-msvc/
│   └── wasm/                   # wasm-pack/wasm-bindgen (or napi wasm32-wasip1-threads)
│       └── Cargo.toml  src/lib.rs
├── xtask/                      # build system as Rust (codegen, dist, release)
│   └── Cargo.toml  src/main.rs
├── fixtures/                   # .gd corpus + golden trees (data-driven parser tests)
│   ├── parser/                 # *.gd input + *.ast golden trees (RA-style)
│   └── ide/                    # completion/diagnostics scenario fixtures
├── vendor/godot/               # committed extension_api.json (Godot-sync target)
├── tests/                      # cross-crate integration tests
├── benches/                    # criterion benchmarks
├── fuzz/                       # cargo-fuzz parser targets
├── docs/                       # mdBook guide (book.toml + src/SUMMARY.md)
├── LICENSE-APACHE  LICENSE-MIT  THIRD-PARTY-NOTICES.md
├── CONTRIBUTING.md  CODE_OF_CONDUCT.md  SECURITY.md  GOVERNANCE.md  SUPPORT.md
└── .github/
    ├── workflows/

    │   ├── release-plz.yml     # crate Release-PR + crates.io publish
    │   ├── release-napi.yml    # cross-platform native npm matrix (oxc-style)
    │   ├── release-wasm.yml    # wasm npm package
    │   ├── docs.yml            # mdBook → GitHub Pages
```

Rationale recap: flat `crates/` with folder==crate names (matklad); `xtask/` as the Rust build system (rust-analyzer, cargo-xtask); `bindings/{node,wasm}` mirrors swc's `bindings/`; `bindings/node/npm/*` is the napi-rs per-platform package set; data-driven `fixtures/parser/` golden trees follow rust-analyzer's parser-test convention.

## 2. CI strategy (GitHub Actions)

### 2.1 The napi-rs reference matrix (real workflow)

The canonical cross-compile-and-publish reference is the napi-rs package-template CI — <https://github.com/napi-rs/package-template/blob/main/.github/workflows/CI.yml>. Its build matrix (verified):

| Host runner | Target triple | Build approach |
|---|---|---|
| `macos-latest` | `x86_64-apple-darwin` | native |
| `macos-latest` | `aarch64-apple-darwin` | native |
| `windows-latest` | `x86_64-pc-windows-msvc` | native |
| `windows-latest` | `aarch64-pc-windows-msvc` | native |
| `ubuntu-latest` | `x86_64-unknown-linux-gnu` | `--use-napi-cross` (zig) |
| `ubuntu-latest` | `x86_64-unknown-linux-musl` | `-x` (cargo-zigbuild) |
| `ubuntu-latest` | `aarch64-unknown-linux-gnu` | `--use-napi-cross` |
| `ubuntu-latest` | `aarch64-unknown-linux-musl` | `-x` |
| `ubuntu-latest` | `armv7-unknown-linux-gnueabihf` | `--use-napi-cross` |
| `ubuntu-latest` | `wasm32-wasip1-threads` | napi wasm build |

It caches with `actions/cache` keyed on target+host, sets up zig via `mlugg/setup-zig@v2` + `taiki-e/install-action` for `cargo-zigbuild`, uses `docker/setup-qemu-action` for ARM test emulation, runs Linux tests in Node Docker containers (slim + Alpine), and publishes to npm with provenance when the commit matches a semver tag.

The production-grade version is oxc's reusable workflow `reusable_release_napi.yml` (~18 targets incl. s390x, ppc64le, riscv, FreeBSD) — <https://github.com/oxc-project/oxc/tree/main/.github/workflows>; a clean single-file variant is oxc-resolver's `release-napi.yml` — <https://github.com/oxc-project/oxc-resolver/blob/main/.github/workflows/release-napi.yml>. swc by contrast still uses **Docker-image-per-Linux-target** in `publish-node.yml` — heavier than the zig path; prefer the oxc/napi approach for a new library.

### 2.2 Per-job notes

- **fmt / clippy:** `cargo fmt --all -- --check` and `cargo clippy --all-targets --all-features -- -D warnings` (deny on any warning). Gate the rest of CI behind `fmt` via `needs:`.
- **Test matrix:** `cargo test --workspace` on `ubuntu-latest` / `macos-latest` / `windows-latest`. napi-rs additionally matrixes Node versions (e.g. 20/22/24) for the binding tests.
- **MSRV:** pin `rust-version` under `[package]` (Cargo refuses older toolchains) — <https://doc.rust-lang.org/cargo/guide/continuous-integration.html>. Dedicated job installs exactly that toolchain via `dtolnay/rust-toolchain` and runs `cargo check` (or `cargo hack check --rust-version`). tokio does this with a `minrust` job and the MSRV in an env var — <https://github.com/tokio-rs/tokio/blob/master/.github/workflows/ci.yml>. Optionally `cargo-msrv verify` to assert the manifest value — <https://github.com/foresterre/cargo-msrv/blob/main/README.md>.
- **Coverage:** `cargo-llvm-cov` (`taiki-e/install-action@cargo-llvm-cov` → `cargo llvm-cov --all-features --workspace --lcov --output-path lcov.info` → `codecov/codecov-action@v5`) — <https://github.com/taiki-e/cargo-llvm-cov/blob/main/README.md>. Use `--codecov` instead of `--lcov` for region coverage.
- **Caching:** `Swatinem/rust-cache@v2` in every Rust job; install the toolchain *before* it (the key hashes the toolchain + `.cargo/config.toml`) — <https://github.com/Swatinem/rust-cache/blob/master/README.md>.
- **wasm:** either wasm-pack (`wasm-pack build --target {bundler|web|nodejs}`, wraps wasm-bindgen) — <https://github.com/rustwasm/wasm-pack/blob/master/README.md> — or reuse the napi addon via `wasm32-wasip1-threads` (what oxc ships) — <https://napi.rs/docs/concepts/webassembly>. See `08-wasm-web-and-bindings.md` for the decision.

**Cross-compile tooling — when to use which:** napi-rs CLI `--use-napi-cross` (zig, Linux gnu/arm) / `-x` (musl) is the default — <https://napi.rs/docs/cross-build.en>, <https://napi.rs/blog/announce-v3>; cargo-zigbuild for pinned-glibc + macOS `universal2` — <https://github.com/rust-cross/cargo-zigbuild/blob/main/README.md>; cross-rs (Docker, 40+ targets, zero setup) — <https://github.com/cross-rs/cross/blob/main/README.md>; cargo-xwin only if you avoid native `windows-latest` runners.

### 2.3 Recommended CI matrix table

| Job | Runner OS | Rust target | Purpose |
|---|---|---|---|
| `fmt` | `ubuntu-latest` | stable | `cargo fmt --all --check` (gate; others `needs: fmt`) |
| `clippy` | `ubuntu-latest` | stable | `cargo clippy --all-targets --all-features -- -D warnings` |
| `test (linux)` | `ubuntu-latest` | `x86_64-unknown-linux-gnu` | `cargo test --workspace` |

## 3. Release automation, versioning & changelogs

### 3.1 Tool comparison & role table

| Tool | Ecosystem | Role | Bump? | Changelog? | Publish? | Verdict |
|---|---|---|---|---|---|---|
| **release-plz** <https://release-plz.dev/docs/config> | Rust | Release-PR: bumps `Cargo.toml`, changelog, tag, `cargo publish` | yes (Conv. Commits + `cargo-semver-checks`) | yes (embeds git-cliff) | crates.io | **ADOPT** — Rust engine |
| **Changesets** <https://github.com/changesets/changesets> | JS/npm | "Version Packages" PR from `.changeset/*.md` | yes | yes | npm | **ADOPT** — npm engine (Biome's choice) |
| **cargo-dist / dist** <https://github.com/axodotdev/cargo-dist> | Rust→multi | Cross-platform binary artifacts + installers on GitHub Releases | no | no | GH Releases | **ADOPT IF** shipping a standalone CLI |
| **cargo-release** <https://github.com/crate-ci/cargo-release> | Rust | Imperative release w/ hooks | yes (manual level) | wire yourself | crates.io | **FALLBACK** — if you reject Release-PR model |
| **git-cliff** <https://git-cliff.org/docs/> | Any | Changelog from Conventional Commits | no | yes | no | **INDIRECT** — already inside release-plz |
| **Keep a Changelog 1.1.0** <https://keepachangelog.com/en/1.1.0/> | spec | Human changelog format | — | — | — | **ADOPT** as format |
| **SemVer 2.0.0** <https://semver.org/spec/v2.0.0.html> | spec | Versioning contract | — | — | — | **ADOPT** (Cargo 0.x reading) |
| **Conventional Commits 1.0.0** <https://www.conventionalcommits.org/en/v1.0.0/> | spec | Commit/PR-title grammar | — | — | — | **ADOPT** |

Key relationships: release-plz **embeds git-cliff** (don't run it separately); release-plz and cargo-dist are **complementary** (versioning/changelog/crate-publish vs binary artifacts on the Release); **release-plz cannot publish npm** and has **no generic release hook** — its only shell-out points (`commit_preprocessors`, `[changelog].postprocessors`) are changelog-text-only, so npm publishing must be a separate CI step gated on release-plz's `releases_created`/`releases` outputs — <https://release-plz.dev/docs/github/output>.

### 3.2 How swc / oxc / Biome actually do it

| | Rust crates | npm packages | Lockstep? |
|---|---|---|---|
| **oxc** | custom **`cargo-release-oxc`** (NOT release-plz) reading `oxc_release.toml`; git-cliff-core changelog | same tool bumps `package.json`; napi-rs + pnpm publish, OIDC `--provenance` | **partial** — 3 independent product tracks (`crates`, `oxlint`, `oxfmt`), lockstep *within* a track |
| **Biome** | **manual** `cargo publish --workspace` via `workflow_dispatch`; independent 0.x versions, hand-edited | **Changesets** (`@changesets/cli` + `@changesets/changelog-github`); a `fixed` group locks the binary + platform + wasm packages | **no** — independent; only coupling is the npm version compiled in as `BIOME_VERSION` |
| **swc** | custom **`cargo-mono`** + git-cliff; triggered by a `chore: Publish` commit | `pnpm publish -r` via manual `workflow_dispatch` | **no** — separate triggers/sources |

Citations: oxc — `oxc_release.toml` <https://raw.githubusercontent.com/oxc-project/oxc/main/oxc_release.toml>, workflows <https://github.com/oxc-project/oxc/tree/main/.github/workflows>; **oxc does NOT use release-plz** (a common secondary-source error — the only `release-plz.toml` lives in the separate `cargo-release-oxc` tool's own repo). Biome — `.changeset/config.json` <https://raw.githubusercontent.com/biomejs/biome/main/.changeset/config.json>, CONTRIBUTING ("This repository uses changesets to automate the releases of Biome's binaries, the JavaScript libraries and the creation of the CHANGELOG.md…") <https://github.com/biomejs/biome/blob/main/CONTRIBUTING.md>, manual `publish-crates.yml` <https://raw.githubusercontent.com/biomejs/biome/main/.github/workflows/publish-crates.yml>, versioning policy <https://biomejs.dev/internals/versioning/>. swc — `cargo-mono` <https://github.com/kdy1/cargo-mono>, `publish-crates.yml`/`publish.yml` <https://raw.githubusercontent.com/swc-project/swc/main/.github/workflows/publish-crates.yml>.

**Reading:** the realistic models are **Biome** (Changesets npm + manual cargo, independent versions) or **oxc** (one custom tool, lockstep-within-track). swc's bespoke `cargo-mono` is overkill. Our recommendation = "Biome's split, but the Rust side automated by release-plz instead of manual, and unified to one shared version."

### 3.3 Versioning policy

Adopt **SemVer 2.0.0** with the **Cargo 0.x reading**, applied identically to both registries. This is safe because Cargo and npm interpret `0.x` carets identically: "Cargo uses the convention that only changes in the left-most non-zero component are considered incompatible" — <https://doc.rust-lang.org/cargo/reference/semver.html>; npm's `^0.2.3` resolves to `>=0.2.3 <0.3.0`. So one version number behaves the same to consumers on either registry.

- **Single shared version** across all crates + npm packages (lockstep), kept in sync by §3.4. A unified analyzer surfaced through both registries is one product; independent crate versions (swc-style) add bookkeeping you don't need yet.
- **Start at `0.1.0`** (SemVer FAQ). During 0.x: `0.Y.0` (minor) = **breaking**; `0.y.Z` (patch) = compatible feature/fix. Avoid `0.0.z` (every change is breaking).
- **Conventional-Commit → bump mapping:** `fix` → patch, `feat` → patch *while in 0.x* (set release-plz so `feat` doesn't force a 0.x minor), `!`/`BREAKING CHANGE` → minor (the 0.x "major"). release-plz computes this automatically and runs `cargo-semver-checks` to catch unmarked API breaks.
- **Reach `1.0.0`** when both the Rust crate API and the napi/wasm surface are stable and depended upon in production; then switch to standard feat→minor / fix→patch / breaking→major.
- **Changelog:** Keep a Changelog 1.1.0 headings (Added/Changed/Deprecated/Removed/Fixed/Security + Unreleased), emitted by release-plz's git-cliff `[changelog]` config; mirror groupings on the npm side via `.changeset/config.json`.

### 3.4 End-to-end release process (crates.io + npm in sync)

Because Changesets has no awareness of `Cargo.toml` and release-plz has no awareness of `package.json`, the glue is a one-line version-sync step. release-plz is the **source of truth**:

1. **Job A — `release-plz release-pr`** on push to `main`: opens/updates the Rust Release PR (bump + git-cliff changelog + `cargo-semver-checks`).
2. **On merge — `release-plz release`:** tags + `cargo publish` (crates.io **Trusted Publishing / OIDC**, no token — <https://release-plz.dev/docs/github/quickstart>); exposes `releases`/`releases_created`.
3. **Job B (gated `if: …releases_created == 'true'`):** sync version into each `package.json`, then build napi + wasm and `npm publish --provenance` (OIDC, like oxc):
   ```bash
   v=$(cargo metadata --format-version 1 --no-deps | jq -r '.packages[0].version')
   npm version "$v" --no-git-tag-version   # in each bindings/*/npm package dir
   ```
4. **Optional Job C — cargo-dist:** the release tag triggers cargo-dist's generated `release.yml` to build CLI binaries/installers and attach them to the GitHub Release.
5. **One git tag per release** (release-plz `v<version>`); cargo-dist keys off it.

Contributors still add a `.changeset/*.md` for user-facing npm changes (Changesets reads intent files, not commit messages) — Biome enforces this dual requirement. *Future consolidation option:* **Sampo** <https://github.com/bruits/sampo> is a Changesets-style tool with native crates.io + npm support in one tool, which would eliminate the sync script (newer/less proven).

## 4. Conventional commits + git flow

### 4.1 Commit / PR-title convention

Adopt **Conventional Commits 1.0.0** — `<type>[optional scope]: <description>` — <https://www.conventionalcommits.org/en/v1.0.0/>.

- **Types:** `feat`, `fix`, `perf`, `refactor`, `docs`, `test`, `build`, `ci`, `chore`, `revert`. Use crate names as scopes (`feat(parser): …`), like oxc.
- **Breaking changes:** `feat(parser)!: …` (the `!`) and/or a `BREAKING CHANGE:` footer.
- **Semver feed:** `fix` → PATCH, `feat` → MINOR, breaking → MAJOR (mapped down one level in 0.x per §3.3).

### 4.2 Branching + PR flow

Adopt **trunk-based development** — <https://trunkbaseddevelopment.com/>: a single protected `main`, short-lived feature branches, **squash-merge with PR title = the conventional commit**. This is effectively what oxc and rust-analyzer do; Biome's `main`/`next` split is only needed once you ship explicitly-versioned releases (defer until ~1.0).

1. **Lint PR titles** with `amannn/action-semantic-pull-request` (Biome uses it) — <https://github.com/amannn/action-semantic-pull-request>.
2. **Squash-merge with "Default to PR title for squash merge commits"** enabled, so the conventional PR title becomes the single `main` commit release-plz reads.
3. **Enable GitHub's native merge queue** to keep `main` always-green — it tests each PR against the merged result including PRs ahead of it, catching semantic conflicts. This is GitHub's first-party version of rust-analyzer's **bors / "Not Rocket Science Rule"**: rust-analyzer "use[s] bors to enforce the [not rocket science] rule" — <https://rust-analyzer.github.io/book/contributing/>; the rule (Graydon Hoare) is to keep a repo that "always passes all the tests" — <https://graydon2.dreamwidth.org/1597.html>; GitHub docs — <https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/configuring-pull-request-merges/managing-a-merge-queue>.
4. **npm caveat:** Changesets reads `.changeset/*.md`, not commits — require a changeset for user-facing npm changes even though the Rust side derives bumps automatically.

**Assembled pipeline:** branch → PR (conventional title) → title lint → CI + merge queue keep `main` green → squash-merge (title→commit) → release-plz Release-PR + Changesets "Version Packages" PR → merge triggers `cargo publish` (OIDC) then version-sync + napi/wasm `npm publish --provenance` → tag triggers optional cargo-dist artifacts.

## 5. Documentation

### 5.1 docs.rs (API docs)

docs.rs auto-hosts rustdoc for every crate published to crates.io — no opt-in; builds in a **network-isolated sandbox** (all docs must build offline). Customize per-crate via `[package.metadata.docs.rs]` (`features`, `all-features`, `targets`, `rustdoc-args`, …) — <https://docs.rs/about/metadata>. Idiomatic feature-label pattern:

```toml
[package.metadata.docs.rs]
all-features = true
rustdoc-args = ["--cfg", "docsrs"]
```

plus `#![cfg_attr(docsrs, feature(doc_cfg))]` at each crate root. Preview: `RUSTDOCFLAGS="--cfg docsrs" cargo +nightly doc --no-deps --all-features`.

### 5.2 mdBook (guide site)

mdBook is the generator behind the Rust Book, rustc/Cargo books, and the rust-analyzer manual. `book.toml` + `src/` Markdown → `mdbook build`; **`SUMMARY.md`** is the single TOC that drives the whole structure — <https://rust-lang.github.io/mdBook/format/summary.html>. CI should run `mdbook test` (validates Rust samples) + `mdbook-linkcheck` — <https://rust-lang.github.io/mdBook/continuous-integration.html>. Deploy via the **official GitHub Pages starter workflow** (`actions/configure-pages` → `mdbook build` → `actions/upload-pages-artifact` → `actions/deploy-pages`, OIDC, no `gh-pages` branch) — <https://github.com/actions/starter-workflows/blob/main/pages/mdbook.yml>.

### 5.3 How comparable tools split docs

Consistent pattern: **hand-written guide/manual (use the tool) is separate from auto-generated rustdoc on docs.rs (call the library)**. rust-analyzer — mdBook manual at <https://rust-analyzer.github.io/book/> (Install/Config/**Other Editors** (LSP-client integration)/Features/Architecture). oxc — `oxc.rs` is VitePress. Biome — `biomejs.dev` is Astro + Starlight (Guides/Reference/Internals/**Playground**). Rust-first projects lean mdBook+docs.rs; JS/npm-flavored ones add VitePress/Astro for playgrounds + i18n. `gdscript-analyzer` straddles both — start mdBook, wrap with a VitePress front site + playground later if wanted.

### 5.4 README + the "add a client" page

README: dense, badges (crates.io version, docs.rs build, CI, npm version, license), quickstart/minimal example, comparison-to-alternatives + limitations, links out to guide + API. `cargo-readme` can generate it from `lib.rs`.

**The "Add an LSP client" page** (model on rust-analyzer's *Other Editors* — <https://rust-analyzer.github.io/book/other_editors.html>) must cover: locating the server binary on `PATH`; transport (stdio/TCP); all settings via LSP `initializationOptions` in `InitializeParams` (document the JSON schema + a default example); per-editor setup blocks (VS Code, Neovim, Godot external editor); advertised LSP capabilities + how to enable tracing for client authors; cross-link to docs.rs for embedding the crates directly.

### 5.5 Recommended docs stack

- **API:** docs.rs with `all-features = true` + `--cfg docsrs`; ensure offline builds.
- **Guide:** **mdBook** deployed via the official Pages workflow; `mdbook test` + `mdbook-linkcheck` in PR CI.
- **SUMMARY.md skeleton:** User Guide (Install/Quickstart/Config/Features) → **Editor/LSP Client Integration** (Overview, VS Code, Neovim, Godot external editor, "Adding a new client") → Reference (CLI/config/capabilities) → Contributing (Architecture/Crate layout/Build).

### 6.2 RFC vs ADR — recommendation

- **Rust RFCs** (`rust-lang/rfcs`) — heavy: `text/0000-*.md`, sub-team review, 10-day FCP, tracking issue — <https://github.com/rust-lang/rfcs>.
- **Bevy RFCs** (`bevyengine/rfcs`) — deliberately lighter: "most changes don't need RFCs," "a tool for collaboration, not a burden" — <https://github.com/bevyengine/rfcs>.
- **ADRs** (Nygard, 5 sections Title/Status/Context/Decision/Consequences, numbered markdown in `docs/adr/`) — <https://github.com/joelparkerhenderson/architecture-decision-record>.

**Bottom line:** adopt **lightweight in-repo ADRs now; do NOT stand up a separate RFC repo yet.** The project is early and foundational — most consequential decisions are internal architecture (IR, query/incremental layering, name resolution, error recovery, crate↔npm/wasm boundary), the sweet spot for cheap Nygard ADRs. Route rare user-facing changes through a `proposal` issue form + an `S-needs-design` label (Bevy philosophy). Graduate to a dedicated `rfcs` repo with FCP only when external contributor volume justifies it.

### 6.3 Security + triage

`SECURITY.md` points at **Private Vulnerability Reporting** (a separate GitHub feature, enable it) — <https://docs.github.com/code-security/getting-started/adding-a-security-policy-to-your-repository>. Run **`cargo-audit`** (RustSec advisory DB, `RUSTSEC-YYYY-NNNN`, reads `Cargo.lock`) via the `rust-audit-check` Action — <https://rustsec.org>. Seed rust-analyzer-style labels: `C-bug/C-enhancement/C-diagnostic/C-architecture`, `E-easy/medium/hard`, `S-actionable/unactionable/needs-repro/needs-info/needs-design`, `good-first-issue`, `broken-window` — <https://rust-analyzer.github.io/book/contributing/>; milestones per release.

## 7. Licensing

### 7.1 Dual `MIT OR Apache-2.0` (the Rust norm)

The Rust project and all official projects are dual-licensed Apache-2.0 + MIT — <https://www.rust-lang.org/policies/licenses>. **Why both:** Apache-2.0 brings an explicit **patent grant** (and clearer derivative-work terms) MIT lacks; MIT preserves **GPLv2 compatibility** (Apache-2.0 is GPLv2-incompatible) and max permissiveness. Offering *either* lets downstream users pick Apache patent protection or the MIT/GPLv2 path.

**Mechanics** (Rust API Guidelines "Necessities" explicitly recommends this — <https://rust-lang.github.io/api-guidelines/necessities.html>): workspace `Cargo.toml` `license = "MIT OR Apache-2.0"` (SPDX `OR` is load-bearing), inherit via `license.workspace = true`; ship `LICENSE-APACHE` + `LICENSE-MIT` in root; README license section. **`[C-PERMISSIVE]`:** a permissive crate should only depend on permissive crates — audit every dep with `cargo-deny`. npm side: `"license": "(MIT OR Apache-2.0)"` (parentheses required) + include both license files in the tarball.

### 7.2 Ingested inputs — both MIT, attribution-only

- **Godot** is **MIT (Expat)** — <https://godotengine.org/license>. `extension_api.json` is generated output of the MIT engine (`godot --dump-extension-api`); the JSON doesn't embed license text, so attribute it externally. **Precedent: godot-cpp** (official C++ bindings) is MIT and ingests `extension_api.json` — <https://github.com/godotengine/godot-cpp>. Prefer generating/fetching per target Godot version (see §8); vendoring is fine under MIT if attributed.
- **tree-sitter-gdscript** (canonical: <https://github.com/PrestonKnopp/tree-sitter-gdscript>) is **MIT**. **⚠️ Gotcha:** its `LICENSE` retains the unchanged tree-sitter template copyright line — **"Copyright (c) 2016 Max Brunsfeld"** (not Preston Knopp). If you bundle the grammar, retain that notice **verbatim** and credit the repo in prose. This default-template situation is common across tree-sitter grammars — read each grammar's actual LICENSE.

### 7.3 Compatibility + NOTICE approach

Both inputs are MIT; MIT → MIT/Apache-2.0 dual is one-way compatible (MIT is a near-subset of Apache-2.0). Bundling MIT Godot data + MIT grammar in an `MIT OR Apache-2.0` crate = **no conflict**, satisfies `[C-PERMISSIVE]`. The only obligation is **notice retention** (MIT requires the copyright + permission notice; Apache §4(d) propagates NOTICE attributions). No copyleft, no relicensing constraint.

**Recommendation:** ship a **`THIRD-PARTY-NOTICES.md`** listing Godot (full MIT copyright + note that `extension_api.json` is MIT-derived Godot output) and tree-sitter-gdscript (retain `Copyright (c) 2016 Max Brunsfeld` verbatim + credit the repo) and the tree-sitter runtime/`parser.c` if bundled. Automate crate attributions with `cargo-about`/`cargo-bundle-licenses` in CI; hand-maintain the two non-crate entries.

## 8. The Godot-sync automation (CI/automation mechanics)

This section covers the **CI plumbing**; the *meaning* of the API delta and the codegen it feeds are in `03-godot-api-sync.md`.

### 8.1 Triggers

- **`schedule` (cron)** — 5-field POSIX cron in `on.schedule.cron`. Four load-bearing caveats (GitHub "Events that trigger workflows" — <https://docs.github.com/en/actions/using-workflows/events-that-trigger-workflows>): (1) runs only on the **latest commit of the default branch** (must be merged first); (2) **5-minute minimum**; (3) **delayed under load** — schedule off-peak (`:17`, not `:00`); (4) **auto-disabled after 60 days of repo inactivity**.
- **`workflow_dispatch`** — manual/API/CLI trigger with inputs (allow a `godot_tag` override).
- **`repository_dispatch`** — external trigger via REST API (`event_type` + `client_payload`); also default-branch-only. `schedule` + `workflow_dispatch` suffices here.

### 8.2 `gh` CLI + create-pull-request

Auth via `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}` (default token is read-only since Feb 2023 — elevate via `permissions:`). Resolve the tag: `gh api repos/godotengine/godot/releases/latest --jq .tag_name`; for stable-only, `gh release list -R godotengine/godot --exclude-pre-releases --exclude-drafts --limit 1 --json tagName --jq '.[0].tagName'` (Godot publishes many `-dev/-beta/-rc` prereleases) — <https://cli.github.com/manual/gh_release_list>. Open the PR with **`peter-evans/create-pull-request`** (latest v8; pin `@v8`) — idempotent, a stable branch updates the existing PR — <https://github.com/peter-evans/create-pull-request>. Requires `permissions: contents: write` + `pull-requests: write` **and** the repo setting *Settings → Actions → General → "Allow GitHub Actions to create and approve pull requests."* (Caveat: PRs from the default `GITHUB_TOKEN` don't trigger downstream CI — use a PAT/App token if the bot's PR must run CI.)

**Sourcing `extension_api.json`:** (A) download the Godot release binary + `godot --headless --dump-extension-api`, or (B, simpler) pull the committed copy from `godotengine/godot-cpp` at the matching tag (`gdextension/extension_api.json`) — no binary needed. The skeleton uses B with A as a commented fallback.

### 8.3 Workflow skeleton

```yaml
# .github/workflows/godot-api-sync.yml
name: Sync Godot extension_api.json

on:
  schedule:
    - cron: '17 6 * * *'        # daily 06:17 UTC (off-peak); default-branch only; auto-disabled after 60d inactivity
  workflow_dispatch:
    inputs:
      godot_tag:
        description: 'Override Godot tag (e.g. 4.4.1-stable). Empty = latest stable.'
        required: false
        type: string

permissions:
  contents: write
  pull-requests: write

concurrency:
  group: godot-api-sync
  cancel-in-progress: false

jobs:
  sync:
    runs-on: ubuntu-latest
    env:
      GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
      COMMITTED_PATH: vendor/godot/extension_api.json
    steps:
      - uses: actions/checkout@v7

      - name: Resolve latest Godot stable tag
        id: godot
        run: |
          set -euo pipefail
          if [ -n "${{ inputs.godot_tag }}" ]; then
            TAG="${{ inputs.godot_tag }}"
          else
            TAG="$(gh api repos/godotengine/godot/releases/latest --jq .tag_name)"
          fi
          echo "tag=$TAG" >> "$GITHUB_OUTPUT"

      - name: Fetch upstream extension_api.json
        run: |
          set -euo pipefail
          TAG="${{ steps.godot.outputs.tag }}"
          URL="https://raw.githubusercontent.com/godotengine/godot-cpp/${TAG}/gdextension/extension_api.json"
          curl -fsSL "$URL" -o new_extension_api.json
          # Fallback: download Godot release binary + ./Godot --headless --dump-extension-api

      - name: Diff against committed copy
        id: diff
        run: |
          set -euo pipefail
          mkdir -p "$(dirname "$COMMITTED_PATH")"
          if [ ! -f "$COMMITTED_PATH" ]; then
            cp new_extension_api.json "$COMMITTED_PATH"
            echo "changed=true" >> "$GITHUB_OUTPUT"
            echo "summary=Initial import for ${{ steps.godot.outputs.tag }}" >> "$GITHUB_OUTPUT"
            exit 0
          fi
          if diff -q "$COMMITTED_PATH" new_extension_api.json >/dev/null; then
            echo "changed=false" >> "$GITHUB_OUTPUT"
          else
            DELTA="$(diff -u "$COMMITTED_PATH" new_extension_api.json | head -n 200 || true)"
            cp new_extension_api.json "$COMMITTED_PATH"
            echo "changed=true" >> "$GITHUB_OUTPUT"
            { echo "summary<<EOF"; echo '```diff'; echo "$DELTA"; echo '```'; echo "EOF"; } >> "$GITHUB_OUTPUT"
          fi

      - name: Regenerate codegen from new API  # see 03-godot-api-sync.md
        if: steps.diff.outputs.changed == 'true'
        run: cargo xtask codegen-godot-api

      - name: Open / update PR
        if: steps.diff.outputs.changed == 'true'
        uses: peter-evans/create-pull-request@v8
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          branch: chore/godot-api-sync          # stable branch => updates existing PR
          base: master
          commit-message: "chore: sync extension_api.json to Godot ${{ steps.godot.outputs.tag }}"
          title: "chore: sync extension_api.json to Godot ${{ steps.godot.outputs.tag }}"
          body: |
            Automated sync to **Godot ${{ steps.godot.outputs.tag }}** (source: godot-cpp @ same tag).
            ## API delta
            ${{ steps.diff.outputs.summary }}
          labels: |
            dependencies
            godot-sync
          delete-branch: false
```

Pinned: `actions/checkout@v7`, `peter-evans/create-pull-request@v8`. Enable the "Allow GitHub Actions to create and approve pull requests" repo setting; swap to a PAT/App token if the bot PR must trigger CI. The `cargo xtask codegen-godot-api` step regenerates the binding/stub data so the PR carries both the raw `extension_api.json` delta *and* the regenerated code for review.

### Confidence flags / verify-before-pinning

- **oxc has no top-level `wasm/` folder** at HEAD; wasm comes from its napi crate (`wasm32-wasi`). Treat older `@oxc-parser/wasm` references as stale.
- **release-plz cannot publish npm** and has **no generic release hook** — npm publishing must be a separate gated CI step. Single most important constraint for the polyglot setup.
- **oxc does NOT use release-plz** (a common secondary-source error) — it uses the homegrown `cargo-release-oxc`. Don't model your setup on that false premise.
- Exact pinned action versions inside the oxc/swc workflows, swc's `crates/` count, oxc's precise merge-queue policy, and a couple of GitHub-UI placement details (CoC template chooser, SECURITY.md placement) came from search snippets rather than full fetches — verify those narrow points before adopting verbatim.
