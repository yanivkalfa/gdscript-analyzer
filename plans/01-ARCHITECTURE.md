# 01 — Architecture

> Canonical technical reference. Fixes the crate layering, the public API shape, the FFI/WASM strategy, the incremental-computation decision, the data model, and the portability rules. Every phase doc obeys this.

Evidence base: [`research/06-analyzer-architecture.md`](research/06-analyzer-architecture.md) (layering, `AnalysisHost`/`Analysis`, FFI), [`research/01`](research/01-rust-distribution-tooling.md) (binding crates), [`research/02`](research/02-parsing-strategy.md) (parser), [`research/08`](research/08-wasm-web-and-bindings.md) (portability), [`research/09`](research/09-type-system-and-inference.md) (semantic layer).

## 1. The layered crate stack

We copy rust-analyzer's proven layering: **each crate depends only downward**, the lower crates know nothing about LSP or FFI, and the analysis API boundary is plain data + byte offsets.

```
┌─────────────────────────────────────────────────────────────────────┐
│  CLIENTS (separate crates / packages; each maps POD → its protocol)  │
│  gdscript-lsp   gdscript-cli   @gdscript-analyzer/* (napi+wasm)      │
│  guitkx adapter (lives in the guitkx repo, consumes the npm pkg)     │
└───────────────▲─────────────────────────────────────────────────────┘
                │ depends on
┌───────────────┴───────────────┐   gdscript-ffi  (napi-rs v3 → Node .node + wasm32;
│         gdscript-ide          │◀──   thin: JSON in / JSON out; holds an AnalysisHandle)
│  AnalysisHost / Analysis;     │
│  features: completion, hover, │
│  goto, refs, rename, symbols, │
│  semantic-tokens, inlay-hints,│

                │

                │
```

### Crate responsibilities

| Crate | Responsibility | Depends on | wasm-safe? |
|---|---|---|---|
| `gdscript-base` | `FileId`, `TextSize`/`TextRange`, `LineIndex`, position/range conversions, the serde POD result types shared with clients. | — | ✅ |
| `gdscript-syntax` | Lexer + indentation pre-pass + parser → **lossless CST** (`cstree`) + typed AST layer. Error recovery. The `Parser` trait + backends. | base | ✅ |
| `gdscript-api` | The Godot engine model loaded from generated artifacts (`extension_api.json` → rkyv/serde): classes, inheritance chain, methods, properties, signals, enums, constants, singletons, utility functions, builtin types. Plus the hand-authored GDScript layer (keywords, annotations, builtins) the dump omits. | base | ✅ |
| `gdscript-db` | The input layer: a virtual file system (`FileId` → text, injected — **never** `std::fs`), the project model, `apply_change`. MVP: plain maps. v1: **salsa** inputs + tracked queries. | base, syntax, api | ✅ |
| `gdscript-hir` | The semantic layer: lower AST → HIR (ItemTree/DefMap/Body), scope tree, name resolution, **gradual type inference**, the GDScript warning checks. | base, syntax, api, db | ✅ |
| `gdscript-ide` | The **feature** layer and the **public API**: `AnalysisHost` + immutable `Analysis` snapshots; one method per IDE feature, each returning POD. **This is the crate external Rust consumers depend on.** | all above | ✅ |
| `gdscript-scene` | `.tscn`/`.tres` parser; node-tree model for path typing. (Phase 4.) | base | ✅ |
| `gdscript-ffi` | The **only** crate with napi/wasm glue. napi-rs v3 → a Node `.node` addon **and** a `wasm32` build from the same source. Holds an `AnalysisHandle` (keeps the host alive across edits); JSON in / JSON out. | ide | n/a (is the binding) |
| `gdscript-lsp` | A real LSP server binary. Maps LSP ↔ our POD API. The only place that knows `lsp-types`/JSON-RPC. | ide | native |
| `gdscript-cli` | `check`/`lint`/`format`/`symbols` for CI. | ide | native |
| `xtask` | Build automation: regenerate `gdscript-api` data from `extension_api.json`, manage fixtures, dist, release helpers. | — | native |

**Naming note:** internal workspace crate names use the `gdscript-` prefix. Published **crates.io** names may need a uniqueness prefix if taken (verify availability; fall back to `gdscript-analyzer-<layer>`, mirroring rust-analyzer's `ra_ap_*`). The **npm scope** is `@gdscript-analyzer/*` (verify availability) — e.g. `@gdscript-analyzer/core` (napi), `@gdscript-analyzer/wasm`, with per-platform binary sub-packages via `optionalDependencies`.

## 2. The public API — `AnalysisHost` + `Analysis`

Modeled exactly on rust-analyzer's `ide::AnalysisHost` / `ide::Analysis` ([`research/06`](research/06-analyzer-architecture.md)).

```rust
// gdscript-ide (sketch — illustrative, not final)

/// The single mutable owner of analysis state. One per project/workspace.
pub struct AnalysisHost { /* db: RootDatabase */ }

impl AnalysisHost {
    pub fn new() -> Self;
    /// Apply a batch of input changes (file added/edited/removed, project config,
    /// Godot version, scene files). The ONLY mutation entry point.
    pub fn apply_change(&mut self, change: Change);
    /// Cheap, cloneable, immutable, `Send` snapshot for read queries.
    pub fn analysis(&self) -> Analysis;
}

/// An immutable snapshot. Every query is cancellable (a newer change cancels in-flight reads).
pub struct Analysis { /* db: salsa::Snapshot<RootDatabase> */ }

impl Analysis {
    // All positions are byte offsets within a FileId. All results are POD (serde).
    pub fn diagnostics(&self, file: FileId) -> Cancellable<Vec<Diagnostic>>;
    pub fn completions(&self, pos: FilePosition) -> Cancellable<Vec<CompletionItem>>;
    pub fn hover(&self, pos: FilePosition) -> Cancellable<Option<HoverResult>>;
    pub fn signature_help(&self, pos: FilePosition) -> Cancellable<Option<SignatureHelp>>;
    pub fn goto_definition(&self, pos: FilePosition) -> Cancellable<Vec<NavTarget>>;
    pub fn find_references(&self, pos: FilePosition) -> Cancellable<Vec<Reference>>;
    pub fn rename(&self, pos: FilePosition, new: &str) -> Cancellable<Result<SourceChange, RenameError>>;
    pub fn document_symbols(&self, file: FileId) -> Cancellable<Vec<DocumentSymbol>>;
    pub fn workspace_symbols(&self, query: &str) -> Cancellable<Vec<NavTarget>>;
    pub fn semantic_tokens(&self, file: FileId) -> Cancellable<SemanticTokens>;
    pub fn inlay_hints(&self, file: FileId) -> Cancellable<Vec<InlayHint>>;
    pub fn folding_ranges(&self, file: FileId) -> Cancellable<Vec<FoldRange>>;
    pub fn code_actions(&self, range: FileRange) -> Cancellable<Vec<CodeAction>>;
    pub fn format(&self, file: FileId) -> Cancellable<Option<SourceChange>>;
}
```

**Rules of the API surface:**
- **Inputs:** `FileId` + byte offsets. The host owns a virtual file system; clients push text via `apply_change` (no path reads inside the library).
- **Outputs:** POD structs defined in `gdscript-base`, `serde`-serializable, **no `lsp-types`**. A `Diagnostic` carries a byte `TextRange`, a code (e.g. `GDSCRIPT_UNSAFE_CALL`), a severity, a message, and optional fixes — the LSP client converts ranges to UTF-16 and codes to its shape.
- **Cancellation:** reads return `Cancellable<T>`; a concurrent `apply_change` cancels in-flight reads (salsa panics-and-catches at this boundary). Clients re-issue.
- **Stability:** this is the crate we semver most carefully; it is the contract every consumer builds on.

## 3. Incremental computation — salsa, designed-for-now, adopted-at-Tier-2

[`research/06`](research/06-analyzer-architecture.md) §salsa. Current: **salsa 0.27** (macro-based "new salsa": `#[salsa::input]`, `#[salsa::tracked]`, `#[salsa::interned]`, `#[salsa::accumulator]` for diagnostics, red-green invalidation, durability).

- **MVP (Phase 1–2):** do **not** pull in salsa yet. Use a plain `HashMap<FileId, Arc<Parsed>>` and re-parse/re-analyze on change. GDScript files are small; whole-file recompute is fine for single-file features. **But** write every derived computation as a **pure `(db, file) -> value` function** so the swap to `#[salsa::tracked]` is localized, not a rewrite.
- **Tier 2 (Phase 3):** adopt salsa when cross-file resolution makes "recompute everything per keystroke" untenable. Use **durability** to separate volatile inputs (the file being edited) from durable ones (the Godot stdlib API, unchanged for a session). This preserves the invariant **"editing a function body never invalidates project-global data,"** which is what keeps a project-wide analyzer keystroke-fast.

## 4. The FFI / WASM strategy

[`research/01`](research/01-rust-distribution-tooling.md) + [`research/08`](research/08-wasm-web-and-bindings.md). One core, thin bindings.

- **Primary binding: `gdscript-ffi` on napi-rs v3.** napi-rs v3 compiles the **same** binding source to both a **Node native `.node` addon** *and* a **`wasm32-wasip1-threads`** target — "you don't need to write two different bindings." This serves the Node/LSP path at native speed.
- **Browser playground:** prefer the napi-rs v3 wasm target; keep a **dedicated `crates/wasm` (wasm-bindgen + `wasm-pack build --target web`)** as a documented fallback when we want a smaller artifact with no SharedArrayBuffer/COOP-COEP requirement (Biome/Ruff's approach). Decide per measured bundle size in Phase 5.
- **Shape across the boundary:** a **stateful `AnalysisHandle`** kept in Rust (so the analysis cache survives edits) — the JS side calls `applyChange(...)` and `query(fileId, offset)` and gets **JSON POD** back (serde). Also expose a **stateless one-shot** `analyze(files) -> Report` for CLI/CI. Keep the FFI surface **small and flat** — strings/structs cross the WASM boundary **by copy** (serde-wasm-bindgen), so never return a full AST per call; return only the feature result.
- **Position encoding:** emit byte offsets from the core; the LSP/WASM client converts to **UTF-16** (LSP default) — ship a byte→UTF-16 converter in the client glue (Ruff/Biome both do this). This is a known footgun; bake it into `gdscript-base`'s `LineIndex`.
- **Other languages (cheap optionality, post-v1):** `crates/ffi-c` (cbindgen → C header; reaches Go/C#/Swift/Java) and `crates/py` (PyO3 + maturin → PyPI). No core changes required.

## 5. The data model (engine knowledge)

[`research/03`](research/03-godot-api-sync.md). `gdscript-api` is generated, not hand-maintained:

- **Source of truth:** `extension_api.json` (the `godot --dump-extension-api` output — the same artifact `gdext`/godot-cpp/C# bindings consume). Sections: `header`, `builtin_class_sizes`, `global_constants`, `global_enums`, `utility_functions`, `builtin_classes` (~38 Variant types), `classes` (~1000+ engine classes), `singletons`, `native_structures`.
- **Hover/docs source:** the per-class **doc XML** (`doc/classes/*.xml` + `@GDScript.xml`/`@GlobalScope.xml`), converted BBCode→Markdown — strictly more complete than the JSON `--with-docs`.
- **What the dump is MISSING and we must hand-author** in `gdscript-api`: GDScript keywords; the 36 annotations (`@export*`, `@onready`, `@tool`, `@rpc`, …); GDScript builtins (`preload`, `load`, `range`, `len`, …); and the `@GlobalScope`/`@GDScript` pseudo-classes (doc-XML only). Plus **project-local** symbols (autoloads, `class_name` globals) come from `project.godot`, resolved at runtime, not from the dump.
- **Multi-version:** bundle several Godot minor versions' API; the consumer's project picks one (detected from `project.godot` `[application] config/features`, snapped to the nearest bundled minor, newest as default, override allowed). Generation + diffing is automated — see [`GODOT-SYNC.md`](GODOT-SYNC.md).
- **Browser data shipping:** `extension_api.json` is several MB; for WASM, prune → convert to **rkyv** (zero-copy) or postcard → **brotli** → fetch as a **separate content-hashed asset** (not `include_bytes!`). Lands ~hundreds of KB on the wire.

## 6. The parser model

[`research/02`](research/02-parsing-strategy.md). Decision: **own the parser**, bootstrap pragmatically.

- **End state:** a hand-written, **lossless**, error-recovering recursive-descent parser producing a **`cstree`** CST (Send+Sync + interning, better than `rowan` for our concurrency), with a typed AST layer on top. We control grammar, recovery, and diagnostics — required for a "Roslyn for Godot."
- **Lexer:** `logos`-based, plus a **hand-written indentation pre-pass** that injects INDENT/DEDENT/NEWLINE (an indent stack + bracket-depth counter to suppress significance inside `()[]{}` + backslash line-continuation handling + tab/space rules), modeled on Godot's own tokenizer. This isolates GDScript's Python-like significant-indentation risk into one tested module.
- **Bootstrap:** put a `Parser` trait in front. For a week-1 MVP we *may* wrap **tree-sitter-gdscript** (MIT, mature, has Rust + WASM bindings) behind the trait to get something parsing immediately; then swap in our hand-written backend and **demote tree-sitter to a permanent test oracle** (differential testing against its trees). tree-sitter is the *reference grammar and oracle*, never the grammar-of-record (it's single-maintainer, manually synced to Godot, and models comments lossily).
- **Incremental:** MVP = whole-file reparse (files are small). Later = block-level reparse.

## 7. Portability rules (non-negotiable — WASM correctness)

[`research/08`](research/08-wasm-web-and-bindings.md). The core crates (`base`, `syntax`, `api`, `db`, `hir`, `ide`) **must** compile to `wasm32`:

1. **No `std::fs` / `std::net`** anywhere in the core. File contents are injected through the VFS; the *client* (or the napi binding on native) reads files.
2. **No `std::time::Instant::now()` / `SystemTime::now()`** — they panic on `wasm32-unknown-unknown`. Use injected clocks or `web-time` only in bindings.
3. **No `thread::spawn` in the hot path** — browsers are single-threaded without COOP/COEP + SharedArrayBuffer. Feature-gate any parallelism (`#[cfg(feature = "parallel")]`, native-only).
4. **`getrandom` JS backend** enabled **only** in the wasm binding (feature renamed `wasm_js` in getrandom 0.3+ — a real footgun), never in core.
5. **CI guards this:** `cargo check -p gdscript-ide --target wasm32-unknown-unknown` runs on every PR (rust-analyzer does exactly this).
6. **Avoid `wee_alloc`** (unmaintained); keep default `dlmalloc`, optionally evaluate `talc`.
